    Hi,

At the T2_US_Wisconsin (GLOW) site, we are seeing phedex data transfer (using FTS3 at FNAL/CERN) failures with errors like the following :

"[70] globus_ftp_client: the server responded with an error 421 Idle Timeout: closing control connection."

The phedex xfer job uses "gfal-copy" command (GFAL2 utility) for the copy job, and submitts the job to FTS3  servers either at FNAL/CERN (as configured). Upon failure, the FTS returns the above error message which shows up in the phedex log. Since gfal-copy doesn't provide the name of the actual gridftp server involved (local or remote) it's quite difficult to debug the issue because we have around 350 gridftp servers configured to provide high-throughput data xfer.

I was wondering if "gfal-copy" has a option or could be configured to printout the gridftp server name(s) that was/were involved in the data xfer when the task failed. (That would help debug and pinpoint any issues with that specific server). If not, I think it would be very helpful (in fact really necessary) for the GFAL2 developers to incorporate this feature into the software. (The output from "globus-url-copy" command (with debug option) provides the name of the actual gridftp server which helps tremendously while debugging).

We will appreciate any help you can provide to address this issue.

Thanks,
- Ajit
