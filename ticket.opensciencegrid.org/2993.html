    Jan 18, 2007 04:01 PM UTC by Alan Sill 
    The actual problem is with the way that the OSG template is
distributed.  The problem I had was with the existing version of GUMS
-- John and Jay had responded only because they thought I was having
problems with the new GUMS version, which I was not.

There are several related problems, as we noted later in the ITB call:

1) There is no OSG template distributed with the OSG cache
installation of GUMS.

2) When you pull the file for the OSG template manually after wading
through documentation, it is not a complete working example, but has
to be merged in by hand with pieces of the template that is
distributed with the VDT version of GUMS.

3) There are problems with the Atlas, Dzero and possibly CMS contents:

a) Atlas includes some old entries that cause GUMS to fail
b) Dzero has a dzeroana entry with similar problems
c) the CMS contents do not appear to produce any functional
mapping (though this may be a problem with my own pool mapping setup;
CMS is the only VO that expects pool mapping in its examples, and I
probably do not have it set up properly yet).

So the problem is correctly stated in my original submission and in
the title of the ticket.  The submission could have been clearer in
that these are problems with the OSG GUMS template and pacman
package, not with GUMS itself.

My suggestions are:

- Eliminate the troublesome Atlas and Dzero entries, and/or get them
to resolve them at the VOMS level
- Include a working template (as a template file, not as the default
setup) with the OSG GUMS installation in place of the bare-bones

    template provided by the VDT
- Clarify the steps needed to set up and test pool account mapping
for CMS

By the way, GUMS and PRIMA are a lot of work to set up if only CMS is
going to use them in OSG.  Is there going to be any attempt to unify
progress and practice here?

Thanks,
Alan
