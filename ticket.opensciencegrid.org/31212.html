    Sep 23, 2016 10:12 PM UTC by jazoff@.... 
    The configuration they are using is causing this problem, setting ::/0 to bogus does not properly disable ipv6.

They need to start bind with -4 to completely disable ipv6.

cilogon.org triggers this problem more often than other domains because the name server A records have a lower ttl than the AAAA records.  When ipv6 is not disabled the ns address cache ends up in a state where there is no v4 record but there is a v6 record that has been declared to be bogus, and the next query will fail.

Other domains will show this problem as well with enough testing, as you saw with incommon.org.

with the relevant error being "Failed to sign certificate -- CertificateProviderException  :: From CILogon: Failed to make cilogon/rest request: java.net.UnknownHostException: osg.cilogon.org".

Justin Azoff from CILogon has said:

"All of the cilogon.org related servers are fine, the problem is that there are AAAA records configured for the NS records and bind is not smart enough to realize that ipv6 is not functional before it repeatedly attempts to contact them.
There are DNS records at other times because after an NS server fails to be contacted bind waits UNREACH_HOLD_TIME before trying it again.  This is why the lookup will fail every 10 minutes.
The problem is that your resolvers are still unable to contact the name servers for cilogon.org, most likely because ipv6 is still not fully functional on all resolvers.  If you test it, incommon.org should show the same issues."

However, the python script we are running to collect the information about incommon.org has not produced any results, unlike the script running for CILogon which fails intermittently with "Host osg.cilogon.org not found: 2(SERVFAIL)"
