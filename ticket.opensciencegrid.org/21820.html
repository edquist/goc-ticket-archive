    Sep 10, 2014 03:52 PM UTC by Matyas Selmeci 
    Hi Kevin,

Looks like RSV is still looking for a GRAM Gatekeeper to run the jobs against.

You should make the following changes to files in /etc/osg/config.d on your RSV host:
- Edit 10-gateway.ini and change htcondor_gateway_enabled to True.
- Merge 30-rsv.ini and 30-rsv.ini.rpmnew, in particular looking at the ce_hosts, gram_ce_hosts and htcondor_ce_hosts attributes:

htcondor_ce_hosts should be set to the list of hosts running HTCondor-CE.
htcondor_ce_hosts controls only the HTCondor-CE-specific probes.

gram_ce_hosts should be set to the list of hosts running a Globus Gatekeeper.
gram_ce_hosts controls only the GRAM-specific probes.

ce_hosts should be set to "%(gram_ce_hosts)s,%(htcondor_ce_hosts)s"
ce_hosts controls the CE-specific probes that do not depend on whether you're running GRAM or HTCondor-CE.

After that, rerun osg-configure and try again.

    Hey Kevin,

Here's the document for installing RSV if you plan to deprecate the old instance: https://twiki.grid.iu.edu/bin/view/Documentation/Release3/InstallRSV

If you do not plan to deprecate the previous instance of RSV, you just need to choose which probes you want to enable and run the commands to enable them on the RSV host.

The enable command has this syntax: rsv-control --enable --host host probe-name

So for example, if I for some odd reason wanted to enable srmclient ping against google.com, I would log into the host where I installed RSV and run:

rsv-control --enable --host google.com org.osg.srm.srmclient-ping

    Aug 8, 2014 02:56 PM UTC by OSG-GOC 
    I've done some googling and found the following two pages that seem to have a pretty good overview of what Globus error 3 means:

https://wiki.egi.eu/wiki/Tools/Manuals/TS62
https://twiki.grid.iu.edu/bin/view/Documentation/Release3/GlobusError003

I'll quote the first link just in case there's any problems with people getting to it:

----
Usually caused by lack of memory on the CE where the job was sent to. For example, in /opt/globus/lib/perl/Globus/GRAM/Helper.pm (part of the "lcg" job managers) the queue_submit function calls l_check_memory to check the available memory using the values reported in /proc/meminfo:

my $freefrac = ($memfree+$swapfree)/($memtot+$swaptot);

return 1 if $freefrac < $MIN_MEM_FREE;

The job submission will fail if that ratio falls below $MIN_MEM_FREE, which is 0.2 (i.e. 20%) by default.

As pointed out by Rod Walker, Linux usually keeps $memfree small, so the numerator typically is determined by $swapfree. To avoid the ratio accidentally falling below the threshold, the swap space should be at least as large as the physical memory: in that case the ratio will exceed 0.5 until the swap space starts getting used (not likely when the physical memory is very large).

Error 3 can also occur due to the following causes:

lack of disk space or quota
a permission problem with the grid account home directory
a hardware I/O error
