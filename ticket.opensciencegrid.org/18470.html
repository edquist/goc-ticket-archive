    Jan 10, 2014 05:43 PM UTC by Brian Lin 
    I believe I've tracked down the Condor-G/RSV submission issue to Horst's gahp_server and the globus libraries they're built against. On the submit side, Condor-G starts up a gahp_server, which starts up a gass_server where input files and executables are staged. From there, the job authenticates with the CE's gatekeeper, spins up a job-manager and then uses globus-url-copy to grab the staged files from the gahp_server. The job then times out on the job-manager side because no files are transferred over. I set up 3 scenarios to test where the problem lied:

1) gass_server within a gahp_server on a submit host that had successful Condor-G jobs
2) gass_server within a gahp_server on Horst's submist host with failed Condor-G jobs
3) Standalone gass_server on Horst's submit host

I then tried globus-url-copy using the delegated proxy that gets created on the CE side against all three gass_servers and the only one that failed was #2. #1 and #3 is using up to date globus components out of /usr/lib64 whereas #2 is using the globus libraries that Rocks built Condor against (Horst, correct me if I' wrong here).

The plan of attack at this point is to get an updated version of the gahp_server. Horst doesn't really want to reinstall a completely new version of Condor to do this and Jaime F. said that the gahp_server/Condor interface hasn't changed in a long time so we can just update the gahp_server. I'm not entirely sure what the best method for this would be, build the gahp_server standalone if possible? Grab the binary from another machine?

P.S. This only seems to be a problem when both submit host and CE are using openssl-1.0.1e.
